# Memory Update Protocol — GetKonduit Webpage

---

## What this file is

This file defines when and how to update `memory.md`. It is a procedure document — follow it consistently so memory stays useful across sessions.

---

## Trigger conditions

Update `memory.md` after any session where one of the following occurs:

1. **A meaningful decision was made** — a design choice, copy direction, tech stack decision, or scope change that future-Claude should know about
2. **Something failed or surprised us** — a bug, a broken approach, or feedback that changed direction
3. **A file was created, significantly changed, or deleted** — log it under File Changes so the project map stays accurate

---

## Update procedure

1. Open `memory.md`
2. Add the new entry under the correct section
3. Date every entry: `YYYY-MM-DD — [entry]`
4. Update the `Last updated` field at the top of the file
5. Keep entries concise: one to three sentences max, plus a "So what:" line if the learning has a behavioral implication

---

## Sections in memory.md

| Section | What goes here |
|---------|----------------|
| Learnings | Insights about the project, the client, or the work |
| Open Questions | Unanswered questions that affect decisions |
| Failures | What didn't work and why — no blame, just facts |
| Wins | What worked well and should be repeated |
| File Changes | Files added, deleted, or significantly restructured |
| Patterns | Recurring preferences, shortcuts, or stylistic decisions |
