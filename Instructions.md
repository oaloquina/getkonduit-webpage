# Instructions — GetKonduit Webpage

Five workflows, each with inputs needed and steps. Follow these SOPs for consistent results.

---

## Workflow 1 — Page Design & Code

**Purpose:** Build or update a page on the GetKonduit site.

**Inputs needed:**
- Page name and URL slug (e.g., `/services`, `/about`)
- Design goals or reference sites
- Any existing copy or wireframe

**Steps:**
1. Confirm page purpose: who lands here and what should they do next
2. Scaffold the HTML structure — sections, hierarchy, semantic tags
3. Apply CSS — layout, spacing, typography, responsive breakpoints
4. Wire up any JavaScript (nav, forms, animations)
5. Self-review: does the page look right on mobile and desktop?
6. Save file to workspace and report what was built

---

## Workflow 2 — Copy & Content Writing

**Purpose:** Write or refine copy for a page, section, or asset.

**Inputs needed:**
- Target audience (who is reading this?)
- Page or section (e.g., hero, services section, case study)
- Tone notes (if any) and key messages to hit

**Steps:**
1. Clarify the job of this copy: what action should the reader take?
2. Draft headline, subheadline, body, and CTA
3. Check for clarity: would a first-time visitor understand in under 5 seconds?
4. Check for specificity: replace vague claims with concrete outcomes/benefits
5. Review tone — direct, credible, automation-savvy without being jargon-heavy
6. Deliver copy in the format requested (inline, .md, or injected into HTML)

---

## Workflow 3 — SEO & Performance Audit

**Purpose:** Identify and fix SEO gaps or performance issues on a page or the full site.

**Inputs needed:**
- Page URL or file path to audit
- Target keywords (if known)
- Any PageSpeed or Lighthouse scores already on hand

**Steps:**
1. Scan on-page SEO: title tag, meta description, H1, alt text, canonical tag
2. Check keyword presence and density for target terms
3. Review internal linking structure
4. Assess page load weight: image sizes, scripts, CSS bloat
5. Check accessibility basics: contrast, labels, keyboard nav
6. Deliver a prioritized fix list (quick wins first, then structural issues)

---

## Workflow 4 — Deploy & QA Checklist

**Purpose:** Verify a page or update is ready to go live.

**Inputs needed:**
- File(s) or pages being deployed
- Target environment (local preview, staging, or production)

**Steps:**
1. Links: confirm all internal and external links work, no 404s
2. Forms: test submission flow and confirmation state
3. Mobile: verify layout on narrow viewports (375px minimum)
4. Cross-browser: check Chrome, Safari (and Firefox if possible)
5. Images: confirm all load, have alt text, and aren't oversized
6. SEO: title tag, meta description, and OG tags present
7. Performance: no console errors, no obvious render-blocking issues
8. Sign-off: confirm with user before pushing live

---

## Workflow 5 — Memory Update

**Purpose:** Keep `memory.md` current after meaningful work sessions.

**When to trigger:** See `PROTOCOL.md` for trigger conditions.

**Steps:**
1. Open `memory.md`
2. Add new entries under the relevant sections (Learnings, Failures, Wins, File Changes, Patterns)
3. Update the "Last updated" date at the top
4. Save the file
